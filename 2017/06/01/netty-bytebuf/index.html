netty-bytebuf基础

发表于 2017-06-01 | 分类于 Netty | 阅读次数
| 字数统计

JDK NIO使用了ByteBuffer作为字节容器，操作起来并不方便。
Netty中的ByteBuf则解决了ByteBuf的局限性，使用也更友好。see it…

ByteBuf的API

  • Netty的数据处理AI通过两个组件暴露 – ByteBuf和ByteBufHolder
  • ByteBuf的优点
    • 可以被用户自定义的缓冲区类型锁拓展
    • 通过内置的复合缓冲区类型实现了零拷贝
    • 容量可以按需增长
    • 读写两种模式之间切换不需要调用ByteBuffer的flip()方法
    • 读写方法使用了不同的索引
    • 支持方法的链式调用
    • 支持引用计数
    • 支持池化

ByteBuffer - Netty的数据容器

ByteBuf的工作方式

  • ByteBuf维护了两个不同的索引: 一个用于读取，一个用于写入。当从ByteBuf读取的时候，他的readerIndex将会被递增已经被读取的字节数;同样，当写入的时候，他的writeIndex也会被递增。当readerIndex == writerIndex时，此时就达到了“可以读取的”数据的末尾，试图读取超过该点的数据将会触发IndexOutofBoundException异常。
  • ByteBuf中名字以read和write开头的方法，会不断的更新其对应的索引。而以set或者get开头的操作则不会。可以指定ByteBuf的最大容量，如果试图移动writerIndex超过这个值也会触发异常（默认是Integer.MAX_VALUE）

ByteBuf的使用模式

堆缓冲区

  • 最常用的ByteBuf模式是将数据存储在JVM的堆空间中。这种模式被称为支撑数组（backing array）,能在没有使用池化的情况下提供快速的分配和释放。
  • 适用于有遗留数据需要处理的情况。
  • 当hasArray()方法返回false时，访问支撑数组将会抛出异常。这种使用模式类似于JDK中ByteBuffer的用法。

直接缓冲区

  • JDK NIO的ByteBuffer允许JVM实现通过本地调用来分配内存。避免在每次调用本地IO操作之前或者之后将缓冲区的内容复制到一个中间缓冲区中(Or 相反)。
  • 直接缓冲区的内容将驻留在常规的会被GC回收的堆空间之外，所以直接缓冲区对于网络传输而言是理想的选择。如果数据在一个堆上分配的缓冲区中，那么通过套接字发送之前，JVM将会在内部把你的缓冲区复制到一个直接缓冲区中。当然，直接缓冲区的缺点就是分配和释放都比较昂贵；还有就是涉及到堆上的操作时可能需要数据的复制。

复合缓冲区

  • 为多个ByteBuf提供了聚合视图，可以根据需要增加或者删除ByteBuf实例。Netty通过ByteBuf的子类CompositeByteBuf实现了这个模式。

  • 使用ByteBuffer的复合缓冲模式

  • 使用CompositeByteBuf的复合缓冲区模式
  • 访问CompositeByteBuf中的数据
    • 类似于直接缓冲区模式下的访问。

字节级别的操作

随机索引访问

  • 索引范围: 0 ~ capacity() - 1

顺序访问

  • ByteBuf同时具备读索引和写索引，但是JDK中的ByteBuffer就只有一个索引，所以其必须使用flip()方法在读写模式之间切换的原因。
  • ByteBuf被其两个索引划分为三个区域
    • 0～readerIndex: 已经被读过的可被丢弃的部分
    • readerIndex ~ writerIndex: 尚未被读取的可读部分
    • writerIndex ~ capacity: 可以添加更多字节的可写部分

可丢弃字节

  • 可以通过调用discardReadBytes()来丢弃并且回收空间。初始大小为0, 随着readXXX方法而更新。
  • discardReadBytes()方法可以确保可写分段的最大化，但是这也可能导致内存复制，因为可读字节必须被移动到缓冲区开始的位置(即readerIndex = 0, 可写部分变大)。

可读字节

  • readerIndex默认为0, read或者skip开头的操作都会检索或者跳过位于当前readerIndex的数据，并且将它增加已读字节数。

可写字节

  • 拥有未定义的内容，写入就绪的内存区域。同样，write操作也会修改这个这部分的大小，写入超过目标容量的数据，抛出IndexOutofBoundException异常。

索引的管理

  • 可以通过调用markReaderIndex(), markWriterIndex(), resetReaderIndex()和resetWriterIndex()来标记和重置ByteBuf的readerIndex和writerIndex。
  • 可以通过clear()方法将readerIndex和writerIndex重置为0，但是这不会清空内存中的内容，只是重置索引。

查找操作

  • 普通的查找可以通过indexOf()；复杂的查找需要一个ByteBufProcessor的参数来完成。

派生缓冲区

  • 创建的方法
    • duplicate()
    • slice()
    • slice(int ,int)
    • Unpooled.unmodifiableBuffer()
    • order(ByteOrder)
    • readSlcie(int)
  • 以上的方法都返回一个新的ByteBuf实例，有其自己的读写索引和标记索引，但是其内部存储和原ByteBuf是共享的。
  • ByteBuf的真正复制
    • copy()
    • copy(int ,int )

读写操作

  • get()，set()操作，从给定的索引开始，保存索引不变
  • read()， write(), 给定嗯的index开始，更新index。

ByteBufHolder接口

ByteBufHolder为Netty的高级特性提供了支持，如缓冲区池化。

ByteBuf分配

按需分配

  • Netty通过ByteBufAllocator实现了ByteBuf的池化。
  • 可以通过Channel（每个都可以有一个不同的ByteBufAllocator实例）或者绑定到ChannelHandler的ChannelHandlerContext获取ByteBufAllocator的引用。
  • Netty提供了两种不同的ByteBufAllocator的实现：
    • PoolByteBufAllocator: 池化ByteBuf实例以提高性能，减少内存碎片。默认机制。
    • UnpooledByteBufAlocator: 不池化ByteBuf实例，并且在每次它被调用时都返回一个新的实例。
  • Unpooled的工具类也提供了静态方法来创建未池化的ByteBuf实例。

ByteBufUtil类

  • 操作静态Buf的辅助方法
  • public static String hexDump(ByteBuf buffer) {} // 十六进制打印

引用计数

  • 某个对象持有的资源不再被引用时释放该对象持有的资源。跟踪到某个特定对象的活动引用的数目。当活动引用数目==0,释放。
