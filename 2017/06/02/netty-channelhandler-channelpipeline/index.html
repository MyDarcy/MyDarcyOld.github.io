Netty中的ChannelHandler和ChannelPipeline

发表于 2017-06-02 | 分类于 Netty | 阅读次数
| 字数统计

ChannelPipeline是将ChannelHandler链接到一起以组织处理逻辑。 这里详细了解一下他们，以及表示ChannelPipeline和ChannelHandler关系的组件– ChannelHandlerContext。

ChannelHandler家族

Channel的生命周期

  • Channel接口定义了一组和ChannelInBoundHandler相关的状态模型，包括四个状态
    • ChannelUnregistered: Channel已经创建，但是还没有注册到EventLoop。(Channel在其生命周期内，只能注册到一个EventLoop)
    • ChannelRegistered: Channel已经注册到了EventLoop
    • ChannelActive: Channel处于活动状态（已经连接到远程节点），可以收发数据了。
    • ChannelInactive: Channel没有连接到远程节点
  • Channel正常的生命周期： ChannelRegistered -> ChannelActive -> ChannelInactive -> ChannelUnregistered；当Channel这些状态发生变化的时候，将会生成对应的事件，这些事件将会被转发给ChannelPipeline中的ChannelHandler进行处理。

ChannelHandler的生命周期

  • ChannelHandler接口中定义的的生命周期操作如下， 在ChannelHandler被添加到ChannelPipeline中或者从ChannelPipeline中移除时会调用这些操作，这些方法中的每一个都接受一个ChannelHandlerContext参数。
    • handleAdd: 当把ChannelHandler添加到ChannelPipeline时候调用。
    • handleRemove: 当把ChannelHandler从ChannelPipeline移除的时候调用。
    • exceptionCaught: 处理过程中在ChannelPipeline中有错误产生时被调用。
  • ChannelHandler的两个重要子接口
    • ChannelInBoundHandler – 处理入站以及各种状态的变化
    • ChannelOutBoundHandler – 处理出站并且允许拦截所有的操作。

ChannelInBoundHandler接口

  • ChannelInBoundHandler的生命周期方法如下，这些方法在数据被接收时或者对应的Channel状态发生变化时被调用，这些方法和Channel的生命周期密切相关，参见Channel生命周期。
    • channelRegistered: Channel已经注册到了他的EventLoop并且能够进行IO处理时被调用
    • channelUnregistered: Channel已经从它的EventLoop 注销 并且无法处理任务IO时调用
    • channelActive: Channel处于活动状态被调用；Channel已经连接or绑定并且已经就绪
    • channelInactive: Channel已经不是活动状态并且不再连接它的远程节点时调用
    • channelReadComplete: 所有可读的字节已经从Channel中读取之后，将会回调这个方法，所以这个方法调用之前可能多次调用channelRead()方法。
    • channelRead: 当从Channel中读取数据时调用
    • channelWritabilityChanged: 当Channel的可写状态发生变化时候调用，用户可以确保写操作不会完成的太快，或者可以在Channel变为再次可写时恢复写入，可以调用channel的isWritable()方法检测Channel的可写性。另外，可写阈值也可以进行设置．
    • userEventTriggered: 当ChannelInBoundHandler.fireUserEventTriggered()被调用时候被调用，因为一个POJO已经被传经了ChannelPipeline

释放消息方法1

  • 当某个ChannelInBoundHandler的实现重写了channelRead()方法时候，它将负责显式的释放和池化ByteBuf实例相关的内存。Netty为此提供了一个方法ReferenceCount.release()进行显式的释放．

更简单的做法

  • SimpleChannelInboundHandler会自动的释放资源。所以不要存储任何指向消息的引用以供将来使用，因为这些引用会失效。

ChannelOutBoundHandler接口

  • 出站操作和数据将由ChannelOutboundHandler处理，它的方法将被Channel, ChannelPipeline, ChannelHandlerContext调用
  • ChannelOutboundHandler可以按需推迟操作或者事件，所以就可以通过一些复杂的方法来处理请求。如：到远端节点的写入被暂停了，可以推迟冲刷操作并在稍后继续。
  • ChannelPromise & ChannelFuture: ChannelOutBoundHandler中的大部分方法都需要一个ChannelPromise参数，以便在操作完成以后得到通知。ChannelPromise是ChannelFuture的子类，其中定义了一些可写的方法。如setSuccess()或者setFailure()方法，从而使得ChannelFuture不可变（当一个Promise完成以后，其对应的Future的值就不能再修改了）。

ChannelHandler适配器类

  • ChannelInboundHandlerAdapter 和 ChannelOutboundHandlerAdapter (都拓展了ChannelHandlerAdapter抽象类) 提供了 ChannelInboundHandler interface 和 ChannelOutboundHandler interface 的基本实现。ChannelHandlerAdapter还提供了实用方法isSharable()方法，其对应的实现被标注为Sharable, 那么这个方法将会返回true，表示它可以被添加到多个ChannelPipeline中。
  • ChannelInboundHandlerAdapter 和 ChannelOutboundHandlerAdapter 中提供的方法调用相关联的ChannelHandlerContext上的等效方法，从而将事件转发到ChannelPipeline中下一个ChannelHandler中。

资源管理

  • 通过ChannelInboundHandler.channelRead()或者ChannelOutboundHandler.write()方法来处理数据的时候，需要确保没有任何的资源泄漏(此类并不能自动进行资源的释放)。Netty使用引用计数来处理池化的ByteBuf。所以完全使用完某个ByteBuf之后，调整其引用计数很重要。
  • 为了诊断潜在的资源泄漏问题，Netty提供了ResourceLeakDetector类，它将对应用缓冲区分配做大约1%d的采样来检测内存泄漏．相关的开销很小．4中泄漏检测级别．

消费并释放入站消息

  • 因为ChannelInBoundHandlerAdapter实现ChannelInBoundHandler接口，其并没有进行资源的自动释放．
  • 简单实现: extends SimpleChannelInboundHandler， 这样消息在channelRead0()之后自动的释放。

丢弃并释放出站消息

  • 不仅要释放资源还要通知ChannelPromise，否则可能会出现ChannelFutureListener收不到某个消息已经被处理的通知的情况。
  • 如果一个消息被消费或者丢弃了，并且没有传递给ChannelPipeline中的下一个ChannelOutboundHandler，那么用户就有责任调用ReferenceCountUtil.release()。如果消息达到了实际的传输层，那么当它被写入时或者Channel关闭时，都将被自动释放。

ChannelPipeline接口

  • ChannelPileline可以看作是一个拦截流经Channel的入站和出站事件的Channelhandler实例链，ChannelHandlers之间的交互 –> 应用程序数据和事件的处理逻辑。
  • 每当新创建一个Channel的时候，都将会分配一个新的ChannelPipeline。这个关联是永久性的，不可改变，这时Netty内部完成的，不需要开发人员干预。
  • 事件产生后将会被ChannelInboundHandler或者ChannelOutboundHandler处理，随后，通过调用ChannelHandlerContext的实现，它将被转发给ChannelPipeline中同一超类型（In , Out）的下一个ChannelHandler。ChannelHandlerContext使得ChannelHandler可以和它所属的ChannelPipeline以及其它的ChannelHandler交互。ChannelHandler可以通知其所属的ChannelPipeline中的下一个ChannelHandler，甚至可以动态修改它所属的ChannelPipeline．
  • 在ChannelPipeline中传输事件的时候，它会测试ChannelPipeline中的下一个ChannelHandler的类型是否和事件的运动方向相匹配，直到找到相匹配的ChannelHandler（因为In， Out的区别）

修改ChannelPipeline

  • ChannelPipeline可以通过添加删除，或者替换其他的ChannelHandler来实时的修改ChannelPipeline的布局。也可以将它自己从ChannelPipeline中移除。
  • 通常ChannelPipeline中的每一个ChannelHandler都是通过它的EventLoop（IO线程）来处理传递给它的事件的，所以不要去阻塞这个线程。

ChannelHandlerContext接口

  • ChannelHandlerContext代表了ChannelHandler和ChannelPipeline之间的关联，每当有ChannelHandler添加到ChanenlPipeline中的时候，都会创建ChannelHandlerContext。ChannelHandlerContext的主要功能是管理它所关联的ChannelHandler和在同一个ChannelPipeline中的其他ChannelHandler之间的交互。
  • ChannelHandlerContext中有很多方法Channel和ChannelPipeline中也有，但是：如果调用Channel或者ChannelPipeline中的这些方法，它们将沿着整个ChannelPipeline进行传播，但是调用ChannelHandlerContext上的同名方法，则将从当前所关联的ChannelHandler开始，并且只会传播给位于该ChannelPipeline中的下一个能够处理该事件的ChannelHandler。
  • 方法太多而且和ChannelPipeline重名，这里不列，参考文档 (io.netty.channel.ChannelHandlerContext)
  • ChannelHandlerContext和ChannelHandler的关联是不会改变的，所以缓存对它的引用是安全的。
  • 相对于其他类的同名方法，ChannelHandlerContext中的同名方法将会产生更短的事件流（因为只会向下传递一次），所以可以利用这种特性提供性能。

使用ChannelHandlerContext

从ChannelHandlerPipeline访问Channel

  • channel.write()方法将会导致写入事件从尾端到头部(top -> bottom)地流经ChannelPipeline。

通过ChannelHandlerContext访问ChannelPipeline

  • 以上的两种操作事件流都是一样的，调用Channel或者ChannelPipeline上的write()方法将一直传播事件通过整个ChannelPipeline，但是在ChannelHandler级别上，事件从一个ChannelHandler到下一个ChannelHandler的移动是由ChannelHandlerContext上的调用完成的。所以要想从特定的ChannelHandler开始进行处理，那么需要获取该ChannelHandler之前的ChannelHandler相关的ChannelHandlerContext，这样就可以通过该ChannelHandlerContext触发和它关联的ChannelHandler的下一个ChannelHandler上的相关调用。

ChannelHandler和ChannelHandlerContext的高级用法。

  • 可以通过ChannelHandlerContext获得与之相关的ChannelPipeline，进而操作ChannelPipeline上的ChannelHandlers，可以据此实现复杂的功能，如:通过ChannelHandler添加到ChannelPipeline中来实现到动态的协议切换。

缓存ChannelHandlerContext的引用

  • 因为ChannelHandler可以从属于多个ChannelPipeline（如果ChannelHandler是可共享的化），所以它也可以绑定到多个ChannelHandlerContext实例。用于这个方法的ChannelHandler必须用＠Shareable标注，否则，将其添加到的多个ChannelPipeline时候会引发异常。所以，为了安全的用于多个并发的Channel(连接)，这样的ChannelHandler必须被实现为线程安全。
  • 在多个ChannelPipeline中安装同一个ChannelHandler的一个常见的原因就是跨越多个Channel收集统计信息．

可共享的ChannelHandler

  • 这样的ChannelHandler可以添加到多个ChannelPiple中，同时ChannleHandler不要包含状态，否则需要同步才能在多线程环境下使用。

异常的处理

处理入站异常

如果在处理入站事件的过程中有异常被抛出，那么它将从它在ChannelInboundHandler里被触发的那一点开始流经ChannelPipeline，要想处理这种类型的入站异常。需要在ChannelInBoundHandler中重写方法exceptionCaught()

基本的异常处理

  • 异常会按照入站的方向流动(如入站事件)，所以实现如下逻辑(异常处理)的ChannelInboundHandler通常应该放在ChannelPipeline末尾，这样异常总会被处理无论它发生在ChannelPipeline中的任何位置。

总结

  • ChannelHandler.exceptionCaught()方法的默认实现只是将异常转发到ChannelPipeline中的下一个ChannelHandle。
  • 如果异常到达了ChannelPiple的尾端，它将会被记录为未被处理
  • 要想定义自定义的处理逻辑，需要重写exceptionCaught方法，并决定是否要将异常传播出去。
  • 如果不实现任何处理处理入站异常的逻辑，Netty会通过Warning级别的日志记录该异常到达了ChannelPipeline的末尾，但没有被处理，并尝试释放该异常．

处理出站异常

  • 出站操作的正常完成以及异常的处理，基于以下的机制
    • 每个出站操作都返回一个ChannelFuture。注册到ChannelFuture上的ChannelFutureListener将在操作完成时候被通知该操作是成功还是失败。
    • 几乎所有的ChannelOutboundHandler的方法都带一个ChannelPromise参数。它是ChannelFuture的子类，可以被分配用于异步通知的监听器。ChannelPromise也提供立即通知的可写方法，可以使得操作的状态在ChannelHandler的方法返回给其调用者时便立即被感知到。

添加ChannelFutureListener到ChannelFuture

  • 通过出站操作返回的ChannelFuture上的addListener()方法。最为常用。
