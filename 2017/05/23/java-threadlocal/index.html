ThreadLocal

  • 线程的局部变量，只有当前线程可以访问。
  • ThreadLocal只是起到一个容器的作用，而为每个线程分配不同的对象，需要在应用层面来保证。
  • 仅仅针对SimpleDateFormat线程安全性的解决方案还有： Java8中线程安全的的 DateTimeFormatter

ThreadLocal的实现原理

  • ThreadLocalMap是ThreadLocal类的的静态内部类，同时是定义在Thread内部的成员变量。设置到了ThreadLocal中的数据其实就是设置到ThreadLocalMap中。key是ThreadLocal对象，value就是需要的值。

ThreadLocal.java

Thread.java

ThreadLocal.ThreadLocalMap.java

  • ThreadLocalMap内部是一系列的Entry构成，每一个Entry都是WeakReference
  • ThreadLocal所持有的对象其实是保存在Thread类的ThreadLocalMap变量中；所以只要线程不退出，那么这些对象会一直存在。当线程退出的时候，其实就会执行exit()方法进行一些清理工作。但是如果使用线程池的话，那么这些线程未必会退出，这种情况下，将大对象设置到ThreadLocal中，可能使得系统存在内存泄漏（这里指访问几次不访问后可能出现内存无法被回收，因为一直被线程持有），所以要及时回收对象的话，可以通过ThreadLocal.remove()方法将变量移除；
  • 如果对于ThreadLocal变量，手动设置其为 null, 那么这个ThreadLocal对应的所有线程的局部变量都有可能被回收(参见上面ThreadLocalMap的实现，其存储的每一个对象都是Entry，而Entry又是继承WeakReference，那么当指向ThreadLocal的强引用被设置为null时，ThreadLocalMap的Entry对象中存储的ThreadLocal的弱引用可能会在GC的时候被清理掉)。
