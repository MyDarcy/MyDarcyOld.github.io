BIO, NIO, AIO在网络编程中简单应用

  • Java NIO是New IO的简称，是替代Java IO的新的IO机制。提供了一套不同于Java标准IO的操作机制。NIO与并发无直接相关关系，但是使用NIO 可以大大提高线程的使用效率。
  • Java NIO涉及的内容包括通道(Channel), 缓冲区(Buffer), 文件IO和网络IO
  • 这里主要是基于Socket, NIO和AIO三种方式的echo服务器的实现。

基于Socket的服务器端的多线程模式

  • Echo服务器会读取client的输入，并且将这个输入原样发回给client，为了完成这个功能，服务器还是需要一整套的Socket处理机制。
  • 服务器使用多线程进行处理时的结构示意图
  • 服务器会为每一个client连接启用一个线程，这个新的线程将只会这个client服务。同时，服务器还会使用一个派发线程处理client的连接请求。

MultiThreadEchoServer.java

  • 在可支撑的线程范围内，尽量多的支持客户端的数量
  • 读取client发送过来的内容, 然后原样发回.

EchoClient.java

这种模式很常用，但是缺点在于：它倾向于让CPU进行IO等待。

极端例子

  • 客户端发送请求时有延时, 向服务器发送数据整个过程会持续6s;
  • 而对于服务器而言，每一个请求的处理时间都是6s左右。因为服务器要先读入client的输入，而client的处理速度使得服务器在一次请求中必须等待6s左右。
  • 这里，服务器处理请求的速度慢，不是因为服务器端的负载比较重，而只是因为在等待client端的IO。

使用NIO进行网络编程

  • 使用NIO就可以将上面的网络IO等待时间从业务逻辑处理线程中抽取出来。
  • NIO的重要组件
    • Channel(通道): 类似于流，一个Channel可以和文件或者网络的socket对应。如果Channel对应于Socket，那么往这个Channel中写数据，就类似于向Socket中写入数据。
    • Buffer(缓冲): 往往和Channel一起使用, 作用相当于一个字节数组或者内存区域。数据需要包装成为Buffer的形式才能和Channel交互(读取或者写入)。
    • Selector(选择器): 和Channel密切相关。Channel的SelectableChannel实现表示可以被选择的通道，其可以将自己注册到Selector中。这样，这个Channel就能够被Selector所管理。而一个Selector就可以管理多个SelectableChannel。当SelectableChannel的数据准备好的时候，Selector就会接收到通知，得到那些已经准备好的数据。(基本的工作原理是Channel注册到Selector上，Selector通过轮询的查看各个Channel的状态, 然后对相关的channel进行处理)
  • 一个Selector可以由一个线程进行管理，而一个SocketChannel则可以表示一个client连接，因此就构成了由一个或者少数几个线程，来处理大量的客户端连接的结构。当client端的数据还没有准备好的时候，Selector会处于等待状态(但是用于管理Selector线程是很少的)，但是一旦有任何一个SocketChannel准备好了数据，Selector就能立即得到通知，获取数据进行处理。

NIOServer.java

  • 多线程同步非阻塞模型;
  • 使用了NIO方法后，即使client端发生了迟钝或者网络延迟等现象, 并不会给服务器带来太大的问题。之前6s的耗时现在都是ms级别的。查看后面的输出信息。
  • selector用于处理所有的网络连接，线程池es用于对每一个client进行相应的处理，每一个请求都会委托给线程池中的线程进行实际的处理。

各个函数的详细说明

startServer()

  • Channel也可以像传统的Socket那样阻塞的方式工作，但是这里设置为非阻塞的模式，这种模式下，我们才可以注册Channel感兴趣的事件，并且在数据准备好的时候, 得到必要的通知。
  • 将ServerSocketChannel绑定到Selector并且注册其感兴趣的事件，这样，Selector就能够为这个Channel服务了。当Selector发现ServerSocketChannel有新的客户端连接时，就会通知ServerSocketChannel进行处理。方法register()的返回值是一个SelectionKey，SelectionKey表示一对Selector和Channel的关系, 当Channel注册到Selector上时,就相当于确定了两者之间的服务关系，SelectionKey就是这个契约。 当Selecotr或者Channel被关闭的时候, 他们对应的SeletionKey就会失效。
  • 注意select()方法是一个阻塞方法，如果没有任何数据准备好，就会等待，一旦有数据可读，就会返回。返回值是就绪的SelectionKey的数目。

doAccept(SelectionKey sk)

  • 和socket类似，当有新的client连接到来时，就会生成一个新的Channel产生表示和client通信的通道**。
  • 对应和Client通信的SocketChannel也设置为非阻塞的，即要求系统准备好IO后，再通知线程进行读取或者写入。
  • 需要将新生成的Channel注册到Selector选择器上，并且告诉Selector，现在对OP_READ 读操作感兴趣。这样，当Selector发现这个Channel已经准备好读时，就能够给线程一个通知。

HandleMsg内部类

  • 简单的将数据放到client端的队列中,同时进行必要的业务逻辑处理。
  • 数据处理完毕以后, 就可以将结果回写到client端,因此需要重新注册感兴趣的消息事件，将写操作也作为感兴趣的事件进行提交。这样Channel准备好写入的时候, 就能够通知线程。

doRead(SelectionKey sk)

  • 为了模拟真实的场景，这里使用单独的线程进行数据的处理，而不用阻塞任务派发线程。

doWrite(SelectionKey sk)

  • 之前doAccept中代表Selector和Channel关系的Selectionkey中附加了封装接收的client消息的EchoClient对象。这里针对同一个client， Selectionkey都是同一个。

NIOClient.java

  • Client用之前实现的两种client实现(EchoClient, HeavySocketClient)都没有问题，这里使用NIO的方式来实现client。

AIO进行网络编程

  • NIO在网络操作中，提供了非阻塞的方法，但是NIO的IO行为仍然是同步的。对于NIO来说，我们的业务线程是在IO操作准备好时，得到通知，接着由这个线程自行进行IO操作。但是IO操作本身仍然是同步的。
  • AIO是异步IO的缩写，它不是在IO准备好的时候在通知线程，而是在IO操作已经完成后，再给线程发出通知。因为，AIO是完全不会阻塞的。此时，业务逻辑将变成回调函数，等待IO操作完成后，由系统自动触发。

AIO EchoServer的实现

AIOServer.java

  • 异步IO需要异步通道AsynchronousServerSocketChannel
  • AsynchronousServerSocketChannel.accept()方法实际上做了两件事， 一是发起accept请求，告诉系统可以开始监听端口了，二是注册CompletionHandler实例，告诉系统一旦有客户端前来连接，如果成功连接，那么就去执行其completed()方法，否则failed()方法。accept()方法不会阻塞, 会立即返回。
  • completed()方法执行的时候，表示已经有client成功连接了。
